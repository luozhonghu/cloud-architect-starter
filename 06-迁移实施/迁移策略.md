# 迁移实施 - 迁移策略

## 一、迁移策略概述

### 1.1 迁移模式

#### 常见迁移模式

| 迁移模式 | 说明 | 适用场景 | 优缺点 |
|---------|------|---------|--------|
| 重新托管 | 简单迁移，最小改动 | 快速上云 | 优点：快速、简单<br>缺点：未充分利用云优势 |
| 重新托管 | 简单迁移，最小改动 | 快速上云 | 优点：快速、简单<br>缺点：未充分利用云优势 |
| 平台重构 | 使用托管服务 | 利用云原生服务 | 优点：降低运维成本<br>缺点：需要适配 |
| 重新架构 | 完全重构 | 长期规划 | 优点：充分利用云优势<br>缺点：成本高、周期长 |

### 1.2 迁移策略选择

#### 策略选择矩阵

| 业务要求 | 技术要求 | 时间要求 | 推荐策略 |
|---------|---------|---------|---------|
| 快速上云 | 低 | 紧急 | 重新托管 |
| 优化成本 | 中 | 中等 | 平台重构 |
| 充分利用云 | 高 | 宽松 | 重新架构 |

#### 策略选择决策树

```
是否需要快速上云？
  ├─ 是 → 重新托管
  └─ 否
     是否需要充分利用云优势？
       ├─ 是 → 重新架构
       └─ 否
          是否有云原生服务可用？
            ├─ 是 → 平台重构
            └─ 否 → 重新托管
```

## 二、迁移方法

### 2.1 迁移方法分类

#### 按停机时间分类

| 方法 | 停机时间 | 数据丢失 | 适用场景 |
|------|---------|---------|---------|
| 在线迁移 | 无 | 无 | 关键业务 |
| 在线切换 | 分钟级 | 无 | 重要业务 |
| 离线迁移 | 小时级 | 最小 | 一般业务 |

#### 按迁移方式分类

| 方法 | 说明 | 优势 | 劣势 |
|------|------|------|------|
| 复制粘贴 | 手动复制数据 | 简单、直观 | 易出错、效率低 |
| 工具迁移 | 使用专业工具 | 自动化、高效 | 需要学习成本 |
| 混合迁移 | 工具 + 手动 | 灵活 | 复杂度高 |

### 2.2 在线迁移

#### 迁移原理

```
源环境 → 同步工具 → 目标环境
  ↓
  实时同步
  ↓
切换 DNS
```

#### 迁移步骤

**步骤 1：建立同步**
```bash
# 使用 DTS 建立数据同步
aliyun dts ConfigureSynchronizationJob \
  --SynchronizationJobId dtsxxxxx \
  --SourceEndpoint.Type MySQL \
  --SourceEndpoint.Port 3306 \
  --SourceEndpoint.HostName 192.168.1.10 \
  --DestinationEndpoint.Type MySQL \
  --DestinationEndpoint.Port 3306 \
  --DestinationEndpoint.HostName rm-xxxxx.mysql.rds.aliyuncs.com
```

**步骤 2：全量数据同步**
```bash
# 启动全量同步
aliyun dts StartSynchronizationJob \
  --SynchronizationJobId dtsxxxxx \
  --SynchronizationJobName full-sync
```

**步骤 3：增量数据同步**
```bash
# 启动增量同步
aliyun dts StartSynchronizationJob \
  --SynchronizationJobId dtsxxxxx \
  --SynchronizationJobName incremental-sync
```

**步骤 4：数据验证**
```bash
# 验证数据一致性
checksum_source=$(mysql -h 192.168.1.10 -u root -p -e "CHECKSUM TABLE db.table;")
checksum_target=$(mysql -h rm-xxxxx -u root -p -e "CHECKSUM TABLE db.table;")

if [ "$checksum_source" == "$checksum_target" ]; then
    echo "数据一致"
else
    echo "数据不一致，需要重新同步"
fi
```

**步骤 5：切换 DNS**
```bash
# 更新 DNS 记录
aliyun dns AddDomainRecord \
  --DomainName example.com \
  --RR www \
  --Type A \
  --Value 47.95.12.34
```

#### 优势与劣势

**优势：**
- 无停机时间
- 数据零丢失
- 业务无感知

**劣势：**
- 技术复杂度高
- 成本较高
- 需要专业工具

### 2.3 在线切换

#### 迁移原理

```
源环境 → 预同步 → 目标环境
  ↓
  停止写入
  ↓
  最终同步
  ↓
  切换 DNS
```

#### 迁移步骤

**步骤 1：预同步**
```bash
# 使用 rsync 预同步文件
rsync -avz --delete /data/ user@target:/data/
```

**步骤 2：停止应用**
```bash
# 停止应用服务
systemctl stop app

# 停止数据库（可选）
systemctl stop mysql
```

**步骤 3：最终同步**
```bash
# 最终同步文件
rsync -avz --delete /data/ user@target:/data/

# 最终同步数据库
mysqldump --all-databases | mysql -h target -u root -p
```

**步骤 4：启动应用**
```bash
# 启动应用服务
systemctl start app

# 启动数据库（可选）
systemctl start mysql
```

**步骤 5：切换 DNS**
```bash
# 更新 DNS 记录
aliyun dns UpdateDomainRecord \
  --RecordId 12345678 \
  --Value 47.95.12.34
```

#### 优势与劣势

**优势：**
- 停机时间短（分钟级）
- 技术复杂度适中
- 成本较低

**劣势：**
- 有短暂停机
- 需要选择合适的时间窗口

### 2.4 离线迁移

#### 迁移原理

```
源环境 → 备份数据 → 目标环境
  ↓
  停止服务
  ↓
  恢复数据
  ↓
  启动服务
```

#### 迁移步骤

**步骤 1：备份应用**
```bash
# 备份应用代码
tar -czf app.tar.gz /app/

# 备份配置文件
tar -czf config.tar.gz /etc/app/
```

**步骤 2：备份数据**
```bash
# 备份数据库
mysqldump --all-databases > database.sql

# 备份文件
tar -czf data.tar.gz /data/
```

**步骤 3：传输数据**
```bash
# 传输备份数据到目标环境
scp app.tar.gz user@target:/tmp/
scp config.tar.gz user@target:/tmp/
scp database.sql user@target:/tmp/
scp data.tar.gz user@target:/tmp/
```

**步骤 4：恢复应用**
```bash
# 恢复应用代码
tar -xzf /tmp/app.tar.gz -C /app/

# 恢复配置文件
tar -xzf /tmp/config.tar.gz -C /etc/app/
```

**步骤 5：恢复数据**
```bash
# 恢复数据库
mysql < /tmp/database.sql

# 恢复文件
tar -xzf /tmp/data.tar.gz -C /data/
```

**步骤 6：启动应用**
```bash
# 启动应用服务
systemctl start app

# 验证应用
curl http://localhost:8080/health
```

#### 优势与劣势

**优势：**
- 技术简单
- 成本最低
- 适合小规模迁移

**劣势：**
- 停机时间长
- 数据可能丢失
- 不适合关键业务

### 3.2 数据库迁移工具

#### DTS（数据传输服务）

**功能：**
- 结构迁移
- 全量数据迁移
- 增量数据迁移
- 数据校验

**配置示例：**

```bash
# 1. 创建迁移任务
aliyun dts CreateMigrationJob \
  --MigrationJobName mysql-migration \
  --SourceEndpoint.Type MySQL \
  --SourceEndpoint.Port 3306 \
  --SourceEndpoint.HostName 192.168.1.10 \
  --DestinationEndpoint.Type MySQL \
  --DestinationEndpoint.Port 3306 \
  --DestinationEndpoint.HostName rm-xxxxx.mysql.rds.aliyuncs.com

# 2. 配置迁移对象
aliyun dts ConfigureMigrationJob \
  --MigrationJobId migt-xxxxx \
  --MigrationObject db.*,table.*

# 3. 启动迁移任务
aliyun dts StartMigrationJob \
  --MigrationJobId migt-xxxxx

# 4. 监控迁移进度
aliyun dts DescribeMigrationJobStatus \
  --MigrationJobId migt-xxxxx
```

#### mysqldump（MySQL）

**全量备份：**
```bash
# 备份所有数据库
mysqldump --all-databases > all_databases.sql

# 备份指定数据库
mysqldump --databases db1 db2 > databases.sql

# 备份指定表
mysqldump db1 table1 table2 > tables.sql
```

**增量备份：**
```bash
# 开启 binlog
vi /etc/my.cnf
# 添加
# log-bin=mysql-bin
# binlog-format=ROW

# 查看当前 binlog 文件
mysql -u root -p -e "SHOW MASTER STATUS;"

# 备份 binlog
mysqlbinlog mysql-bin.000001 > incremental.sql
```

**恢复数据：**
```bash
# 恢复全量备份
mysql < all_databases.sql

# 恢复增量备份
mysql < incremental.sql
```

### 3.3 文件迁移工具

#### rsync

**基本用法：**
```bash
# 同步本地目录到远程
rsync -avz /data/ user@target:/data/

# 同步远程目录到本地
rsync -avz user@source:/data/ /data/

# 删除目标目录中多余的文件
rsync -avz --delete /data/ user@target:/data/

# 显示进度
rsync -avz --progress /data/ user@target:/data/
```

**高级用法：**
```bash
# 排除文件
rsync -avz --exclude '*.log' /data/ user@target:/data/

# 限速
rsync -avz --bwlimit=1000 /data/ user@target:/data/

# 使用 SSH 密钥
rsync -avz -e "ssh -i /path/to/key" /data/ user@target:/data/
```

#### OSS Import

**安装：**
```bash
# 下载工具
wget https://gosspublic.alicdn.com/ossimport/2.3.1/ossimport2_linux64.zip
unzip ossimport2_linux64.zip

# 配置任务
cat > job.conf <<EOF
srcType=local
srcPrefix=/data/
destAccessKeyID=your-access-key
destAccessKeySecret=your-secret-key
destEndpoint=oss-cn-hangzhou.aliyuncs.com
destBucket=my-bucket
isSkipExistFile=true
fileParallel=10
taskParallel=5
EOF
```

**执行迁移：**
```bash
# 启动迁移
java -jar ossimport2.jar -c job.conf

# 监控进度
tail -f import.log
```

### 5.1 数据验证

#### 数据完整性验证

```bash
# 对比源端和目标端的数据量
count_source=$(mysql -h source -u root -p -e "SELECT COUNT(*) FROM db.table;")
count_target=$(mysql -h target -u root -p -e "SELECT COUNT(*) FROM db.table;")

if [ "$count_source" == "$count_target" ]; then
    echo "数据量一致"
else
    echo "数据量不一致：源端 $count_source，目标端 $count_target"
fi
```

#### 数据一致性验证

```bash
# 使用 CHECKSUM TABLE
checksum_source=$(mysql -h source -u root -p -e "CHECKSUM TABLE db.table;" | awk '{print $2}')
checksum_target=$(mysql -h target -u root -p -e "CHECKSUM TABLE db.table;" | awk '{print $2}')

if [ "$checksum_source" == "$checksum_target" ]; then
    echo "数据一致"

    echo "数据不一致"
fi
```

## 六、回滚策略

### 6.1 回滚触发条件

| 条件 | 说明 |
|------|------|
| 功能验证失败 | 核心功能无法正常使用 |
| 性能不达标 | 性能低于目标值的 50% |
| 数据不一致 | 关键数据不一致 |
| 出现严重故障 | 系统无法正常运行 |

### 6.2 回滚步骤

#### 步骤 1：停止应用
```bash
# 停止云端应用
aliyun ecs StopInstance --InstanceId i-xxxxx
```

#### 步骤 2：恢复 DNS
```bash
# 恢复 DNS 到本地
aliyun dns UpdateDomainRecord \
  --RecordId 12345678 \
  --Value 192.168.1.10
```

#### 步骤 3：启动本地应用
```bash
# 启动本地应用
systemctl start app
```

#### 步骤 4：验证应用
```bash
# 验证应用功能
curl http://192.168.1.10/health
```

## 七、总结

迁移策略是上云迁移成功的关键，选择合适的迁移方法、工具和计划，可以降低迁移风险，确保迁移成功。

**关键要点：**
1. 根据业务需求选择迁移策略
2. 在线迁移适合关键业务
3. 离线迁移适合一般业务
4. 使用专业工具提高迁移效率
5. 制定详细的迁移计划
6. 充分验证迁移结果
7. 准备回滚方案

**下一步行动：**
- 选择迁移策略
- 准备迁移工具
- 制定迁移计划
