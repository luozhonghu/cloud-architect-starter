# 架构模式 - 微服务架构

## 一、微服务架构概述

### 1.1 什么是微服务架构？

微服务架构（Microservices Architecture）是一种将单一应用程序开发为一组小型服务的方法，每个服务运行在自己的进程中，并使用轻量级机制（通常是 HTTP API）进行通信。

### 1.2 架构对比

#### 单体 vs 微服务
```
单体架构：
┌─────────────────────────────────┐
│         大型单体应用             │
│  ┌───┬───┬───┬───┬───┬───┐    │
│  │U1 │U2 │U3 │O1 │O2 │P1 │    │
│  └───┴───┴───┴───┴───┴───┘    │
│         ↓                      │
│      单个数据库                 │
└─────────────────────────────────┘

微服务架构：
┌─────┐  ┌─────┐  ┌─────┐  ┌─────┐
│服务1│  │服务2│  │服务3│  │服务4│
└──┬──┘  └──┬──┘  └──┬──┘  └──┬──┘
   │        │        │        │
   └────────┴────────┴────────┘
            ↓
      ┌──────────┐
      │ API网关  │
      └──────────┘
```

### 1.3 微服务特点

| 特性 | 说明 |
|------|------|
| **单一职责** | 每个服务只做一件事 |
| **独立部署** | 服务可以独立部署和升级 |
| **技术异构** | 不同服务可以使用不同技术栈 |
| **去中心化** | 每个服务有独立的数据库 |
| **故障隔离** | 一个服务故障不影响其他服务 |
| **弹性扩展** | 根据需求独立扩展特定服务 |

### 1.4 优缺点对比

#### ✅ 优点

| 优点 | 说明 |
|------|------|
| **灵活扩展** | 针对性地扩展高负载服务 |
| **技术自由** | 每个服务可选择最适合的技术 |
| **快速部署** | 小范围部署，风险低 |
| **故障隔离** | 单个服务故障不影响全局 |
| **团队自治** | 小团队负责单个服务 |
| **易于理解** | 代码库更小，更容易理解 |

#### ❌ 缺点

| 缺点 | 说明 |
|------|------|
| **运维复杂** | 需要管理多个服务 |
| **分布式难题** | 分布式事务、服务发现等 |
| **成本更高** | 资源消耗更多 |
| **开发复杂** | 需要处理服务间通信 |
| **测试困难** | 集成测试更复杂 |
| **数据一致性** | 分布式数据一致性挑战 |

### 2.3 服务间通信

#### 同步通信（REST API）
```java
// Feign 客户端
@FeignClient(name = "order-service", path = "/api/orders")
public interface OrderClient {
    
    @GetMapping("/{id}")
    OrderDTO getOrder(@PathVariable Long id);
    
    @PostMapping
    OrderDTO createOrder(@RequestBody OrderDTO order);
}

// 使用
@Service
public class UserService {
    
    @Autowired
    private OrderClient orderClient;
    
    public UserOrderInfo getUserWithOrders(Long userId) {
        User user = getUserById(userId);
        List<Order> orders = orderClient.getOrdersByUserId(userId);
        return new UserOrderInfo(user, orders);
    }
}
```

#### 异步通信（消息队列）
```java
// 发送消息
@Service
public class OrderService {
    
    @Autowired
    private RocketMQTemplate rocketMQTemplate;
    
    public void createOrder(Order order) {
        // 保存订单
        orderRepository.save(order);
        
        // 发送订单创建事件
        OrderCreatedEvent event = new OrderCreatedEvent(order);
        rocketMQTemplate.convertAndSend("order-created-topic", event);
    }
}

// 消费消息
@RocketMQMessageListener(
    topic = "order-created-topic",
    consumerGroup = "payment-service"
)
@Service
public class OrderCreatedConsumer implements RocketMQListener<OrderCreatedEvent> {
    
    @Override
    public void onMessage(OrderCreatedEvent event) {
        // 处理支付
        paymentService.processPayment(event.getOrder());
    }
}
```

## 三、微服务拆分策略

### 3.1 拆分原则

#### DDD（领域驱动设计）
```
根据业务领域边界拆分：
- 用户域：用户、认证、权限
- 商品域：商品、分类、库存
- 订单域：订单、购物车
- 支付域：支付、退款
- 营销域：优惠券、活动
```

#### 数据库拆分
```
共享数据库（不推荐）：
┌────────────────────────────┐
│     数据库（所有表）        │
│  user  order  product      │
└────────────────────────────┘

独立数据库（推荐）：
┌────────┐ ┌────────┐ ┌────────┐
│用户DB  │ │订单DB  │ │商品DB  │
└────────┘ └────────┘ └────────┘
```

### 3.2 拆分步骤

#### 第一步：识别业务边界
```
分析业务流程：
1. 用户注册/登录
2. 浏览商品
3. 加入购物车
4. 创建订单
5. 支付
6. 发货
7. 评价
```

#### 第二步：定义服务接口
```java
// 用户服务接口
public interface UserService {
    User getUser(Long id);
    User register(RegisterRequest request);
    User login(LoginRequest request);
}

// 订单服务接口
public interface OrderService {
    Order getOrder(Long id);
    Order createOrder(CreateOrderRequest request);
    void cancelOrder(Long id);
}
```

#### 第三步：渐进式迁移
```
阶段1：绞杀者模式
- 新功能用微服务
- 旧功能保留在单体

阶段2：核心功能迁移
- 拆分核心服务
- 保持数据同步

阶段3：完全迁移
- 所有功能迁移
- 停止单体应用
```

## 四、数据一致性方案

### 4.1 最终一致性

#### Saga 模式
```
订单创建流程：
1. 创建订单（正向）
2. 扣减库存（正向）
3. 创建支付（正向）

失败补偿：
3. 取消支付（补偿）
2. 恢复库存（补偿）
1. 取消订单（补偿）
```

**Saga 实现：**
```java
@Service
public class OrderSaga {
    
    @Autowired
    private OrderService orderService;
    @Autowired
    private InventoryService inventoryService;
    @Autowired
    private PaymentService paymentService;
    
    @Transactional
    public void createOrderWithSaga(Order order) {
        try {
            // 步骤1：创建订单
            orderService.create(order);
            
            // 步骤2：扣减库存
            inventoryService.deduct(order.getProductId(), order.getQuantity());
            
            // 步骤3：创建支付
            paymentService.create(order.getId(), order.getAmount());
            
        } catch (Exception e) {
            // 补偿
            compensate(order);
            throw e;
        }
    }
    
    private void compensate(Order order) {
        try {
            paymentService.cancel(order.getId());
        } catch (Exception e) {}
        
        try {
            inventoryService.restore(order.getProductId(), order.getQuantity());
        } catch (Exception e) {}
        
        try {
            orderService.cancel(order.getId());
        } catch (Exception e) {}
    }
}
```

#### 事件溯源
```
存储事件而非状态：
事件流：
1. OrderCreated(orderId=123, userId=1, amount=100)
2. OrderPaid(orderId=123, paymentId=456)
3. OrderShipped(orderId=123, trackingId=789)
4. OrderDelivered(orderId=123)

重建状态：
回放事件 → 获取当前状态
```

### 4.2 强一致性

#### 两阶段提交（2PC）
```
协调者         参与者1       参与者2
  │              │             │
  ├─准备───────→│             │
  │              ├─准备─────→ │
  │              │←─同意──────┤
  │←─同意────────┤             │
  │              │             │
  ├─提交───────→│             │
  │              ├─提交─────→ │
  │              │←─完成──────┤
  │←─完成────────┤             │
```

#### TCC（Try-Confirm-Cancel）
```java
@Service
public class PaymentTCC {
    
    // Try：预扣款
    @Transactional
    public boolean tryDeduct(String userId, BigDecimal amount) {
        // 冻结资金
        freezeFund(userId, amount);
        return true;
    }
    
    // Confirm：确认扣款
    @Transactional
    public boolean confirmDeduct(String userId, BigDecimal amount) {
        // 实际扣款
        deductFund(userId, amount);
        return true;
    }
    
    // Cancel：取消扣款
    @Transactional
    public boolean cancelDeduct(String userId, BigDecimal amount) {
        // 解冻资金
        unfreezeFund(userId, amount);
        return true;
    }
}
```

## 六、微服务监控

### 6.1 监控指标

| 维度 | 指标 | 阈值 |
|------|------|------|
| **服务** | QPS、响应时间、错误率 | QPS > 1000, RT > 500ms, 错误率 > 1% |
| **资源** | CPU、内存、磁盘、网络 | CPU > 80%, 内存 > 85% |

| **调用** | 调用链路、成功率 | 成功率 > 99% |

## 十、总结

### 关键要点

1. **不要盲目上微服务**：评估是否真的需要
2. **从简单开始**：单体 → 模块化单体 → 微服务
3. **选择合适工具**：Spring Cloud、Dubbo、K8s
4. **关注服务治理**：注册发现、负载均衡、熔断降级
5. **完善监控**：全链路追踪、日志聚合
6. **数据一致性**：选择合适的一致性方案
7. **成本控制**：资源优化、自动伸缩

### 微服务 ≠ 银弹

微服务架构能解决很多问题，但也会带来新的挑战。选择架构模式时，应该：
- 评估团队能力
- 考虑业务需求
- 权衡成本收益
- 循序渐进演进

**记住：最好的架构是适合当前业务和团队的架构！**
