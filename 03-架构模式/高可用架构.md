# 架构模式 - 高可用架构

## 一、高可用概述

### 1.1 什么是高可用？

高可用（High Availability，简称 HA）是指系统在规定时间内正常提供服务的能力，通常用可用性百分比来衡量。

**可用性等级：**

| 可用性 | 年度停机时间 | 适用场景 |
|--------|------------|---------|
| 99% | 3.65 天 | 非关键业务 |
| 99.9% | 8.76 小时 | 一般业务 |
| 99.95% | 4.38 小时 | 重要业务 |
| 99.99% | 52.56 分钟 | 核心业务 |
| 99.999% | 5.26 分钟 | 金融级业务 |
| 99.9999% | 31.5 秒 | 电信级业务 |

### 1.2 高可用的核心要素

#### 消除单点故障（SPOF）
- 任何单一组件故障不影响服务
- 冗余部署所有关键组件
- 自动故障切换

#### 快速故障恢复
- 自动检测故障
- 自动故障切换
- 快速恢复服务

#### 数据一致性
- 主备数据同步
- 故障切换时数据不丢失
- 最终一致性

### 1.3 高可用设计原则

| 原则 | 说明 | 实践 |
|------|------|------|
| 冗余 | 关键组件部署多个实例 | 主备、集群 |
| 隔离 | 故域隔离，避免级联故障 | 可用区、地域隔离 |
| 自动化 | 自动检测和故障切换 | 健康检查、自动伸缩 |
| 简化 | 简单的架构更容易维护 | 避免过度设计 |
| 监控 | 全面监控，及时发现问题 | 指标、日志、告警 |

## 二、计算层高可用

### 2.1 多实例部署

#### 负载均衡 + 多实例
```
用户 → SLB → ECS-1
           → ECS-2
           → ECS-3
```

**优势：**
- 单台 ECS 故障不影响服务
- 流量自动分发
- 水平扩展

**配置要点：**
- 最少 2 台实例
- 跨可用区部署
- 健康检查配置

### 2.2 跨可用区部署

#### 架构设计
```
             SLB（跨可用区）
          ┌─────────────┐
          ↓             ↓
       AZ1            AZ2
        ↓              ↓
     ECS集群         ECS集群
```

**优势：**
- 单个可用区故障不影响服务
- 99.95% 可用性
- 就近访问

**配置示例（阿里云）：**
```bash
# 创建跨可用区的 SLB
aliyun slb CreateLoadBalancer \
  --RegionId cn-hangzhou \
  --LoadBalancerName cross-az-slb \
  --AddressType internet \
  --VSwitchId vsw-az1,vsw-az2
```

### 2.3 跨地域部署

#### 架构设计
```
       用户（DNS 负载均衡）
          ↙         ↘
      北京地域       上海地域
         ↓             ↓
       SLB           SLB
         ↓             ↓
      ECS集群        ECS集群
         ↓             ↓
      RDS-主         RDS-备
```

**优势：**
- 单个地域故障不影响服务
- 就近访问，降低延迟
- 全球部署

**技术实现：**
- DNS 负载均衡（加权轮询）
- 全局负载均衡（GSLB）
- 跨地域复制

### 2.4 自动故障恢复

#### 健康检查
```
SLB 定期检查后端实例
  ↓
实例健康 → 继续分发流量
实例不健康 → 停止分发，触发自动恢复
```

**健康检查配置：**
- 检查协议：TCP / HTTP / HTTPS
- 检查端口：应用端口
- 检查间隔：2-5 秒
- 超时时间：2-10 秒
- 健康阈值：2-3 次
- 不健康阈值：2-3 次

#### 自动伸缩
```
负载升高 → 自动增加实例
负载降低 → 自动减少实例
实例故障 → 自动替换实例
```

**自动伸缩配置：**
```json
{
  "scalingRule": {
    "metricType": "CPU",
    "threshold": 70,
    "instanceCount": {
      "min": 2,
      "max": 10,
      "default": 2
    }
  }
}
```

## 三、存储层高可用

### 3.1 数据库高可用

#### 主备架构
```
      应用
       ↓
   负载均衡（读写分离）
    ↙       ↘
主库（写）  从库（读）
   ↓         ↓
  数据同步
```

**优势：**
- 主库故障自动切换到从库
- 读写分离，提升读性能
- 主备数据实时同步

**配置要点：**
- 至少一主一备
- 跨可用区部署
- 自动故障切换

**配置示例（阿里云 RDS）：**
```bash
# 创建高可用版 RDS
aliyun rds CreateDBInstance \
  --Engine MySQL \
  --EngineVersion 5.7 \
  --DBInstanceClass rds.mysql.s3.large \
  --DBInstanceStorage 100 \
  --DBInstanceNetType Intranet \
  --VPCId vpc-xxxxx \
  --VSwitchId vsw-xxxxx \
  --ZoneId cn-hangzhou-b \
  --SlaveZoneId cn-hangzhou-e
```

#### 读写分离
```
写请求 → 主库
读请求 → 从库1、从库2、从库3
```

**优势：**
- 提升读性能
- 减轻主库压力
- 提升并发能力

**配置要点：**
- 至少 1 主 2 从
- 负载均衡读请求
- 主从延迟监控

#### 多主架构
```
应用 → 主库1（写）
应用 → 主库2（写）
        ↓
     数据同步
```

**适用场景：**
- 多地域写入
- 高并发写入
- 就近写入

### 3.2 分布式存储高可用

#### 对象存储多副本
```
上传文件 → 自动复制到多个存储节点
```

**优势：**
- 12 个 9 的数据可靠性
- 节点故障自动恢复
- 无需人工干预

**配置：**
- 默认开启多副本
- 跨机房存储
- 自动纠删码

#### 分布式文件系统
```
文件存储（NAS）→ 多副本存储
```

**优势：**
- 高可靠性
- 共享访问
- 挂载简单

### 3.3 缓存高可用

#### Redis 主从 + 哨兵
```
应用 → 哨兵 → 主库
            → 从库1
            → 从库2
```

**优势：**
- 主库故障自动切换
- 读写分离
- 高可用

**配置要点：**
- 至少 1 主 2 从
- 哨兵集群（至少 3 个）
- 自动故障切换

**配置示例（Redis 哨兵）：**
```conf
# sentinel.conf
port 26379
dir /var/redis/sentinel
sentinel monitor mymaster 10.0.1.10 6379 2
sentinel down-after-milliseconds mymaster 30000
sentinel parallel-syncs mymaster 1
sentinel failover-timeout mymaster 180000
```

#### Redis 集群
```
应用 → Redis 集群（分片）
     → Slot 0-5460
     → Slot 5461-10922
     → Slot 10923-16383
```

**优势：**
- 自动分片
- 高可用
- 线性扩展

**配置要点：**
- 至少 6 个节点（3 主 3 从）
- 跨可用区部署
- 自动故障切换

## 四、网络层高可用

### 4.1 负载均衡高可用

#### SLB 主备架构
```
用户 → DNS（主 SLB IP，备 SLB IP）
      → 主 SLB
      → 备 SLB（故障时切换）
```

**优势：**
- SLB 故障自动切换
- 用户无感知
- 高可用

**配置要点：**
- 主备 SLB
- DNS 轮询或健康检查
- 会话保持配置

#### 跨可用区 SLB
```
      DNS → 主可用区 SLB
           → 备可用区 SLB
```

**优势：**
- 可用区级高可用
- 就近访问

### 4.2 DNS 高可用

#### 多 DNS 服务器
```
用户 → DNS 服务器 1
      → DNS 服务器 2
      → DNS 服务器 3
```

**优势：**
- DNS 服务器故障自动切换
- 全球覆盖

#### DNS 负载均衡
```
域名 → 多个 IP 地址
DNS 轮询返回不同 IP
```

**配置示例：**
```
# BIND 配置
$TTL 60
@ IN SOA ns1.example.com. admin.example.com. (
  2024010101
  3600
  600
  86400
  60 )
@ IN NS ns1.example.com.
@ IN NS ns2.example.com.
www.example.com. IN A 1.2.3.4
www.example.com. IN A 5.6.7.8
www.example.com. IN A 9.10.11.12
```

### 4.3 CDN 高可用

#### 多节点部署
```
用户 → 就近 CDN 节点 → 源站
     → 备份 CDN 节点（主节点故障）
```

**优势：**
- CDN 节点故障自动切换
- 全球覆盖
- 就近访问

## 五、应用层高可用

### 5.1 无状态设计

#### 无状态应用
```
应用不存储会话数据
  ↓
任何实例都能处理请求
  ↓
实例可以随意增减
```

**优势：**
- 易于扩展
- 故障切换简单
- 无需状态同步

**实现方式：**
- 会话存储在 Redis
- 使用 JWT Token
- 无状态 API 设计

### 5.2 熔断降级

#### 熔断机制
```
检测到依赖服务异常
  ↓
熔断器打开
  ↓
直接返回降级数据
  ↓
避免级联故障
```

**配置示例（Hystrix）：**
```java
@HystrixCommand(
    fallbackMethod = "fallbackMethod",
    commandProperties = {
        @HystrixProperty(name="circuitBreaker.enabled", value="true"),
        @HystrixProperty(name="circuitBreaker.requestVolumeThreshold", value="20"),
        @HystrixProperty(name="circuitBreaker.sleepWindowInMilliseconds", value="5000")
    }
)
public String callService() {
    // 调用依赖服务
}

public String fallbackMethod() {
    // 降级逻辑
    return "降级数据";
}
```

#### 降级策略
- 关闭非核心功能
- 返回缓存数据
- 返回默认值
- 排队等待

### 5.3 限流

#### 限流策略
- **令牌桶**：固定速率放行请求
- **漏桶**：固定速率处理请求
- **固定窗口**：固定时间窗口限制请求数
- **滑动窗口**：滑动时间窗口限制请求数

**配置示例（Redis + Lua）：**
```lua
-- 固定窗口限流
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local current = tonumber(redis.call('get', key) or '0')
if current + 1 > limit then
    return 0
else
    redis.call('INCRBY', key, 1)
    redis.call('EXPIRE', key, 60)
    return 1
end
```

### 5.4 异步处理

#### 消息队列
```
同步调用 → 消息队列 → 异步处理
```

**优势：**
- 解耦服务
- 削峰填谷
- 提升性能

**配置示例：**
```python
# 使用消息队列异步处理
import pika

connection = pika.BlockingConnection(pika.ConnectionParameters('localhost'))
channel = connection.channel()

# 发送消息
channel.basic_publish(exchange='',
                      routing_key='task_queue',
                      body=message)

# 消费消息
def callback(ch, method, properties, body):
    # 异步处理
    pass

channel.basic_consume(queue='task_queue',
                      on_message_callback=callback)
```

## 六、监控与告警

### 6.1 监控指标

#### 基础指标
- CPU 使用率
- 内存使用率
- 磁盘使用率
- 网络流量

#### 应用指标
- 请求量（QPS）
- 响应时间（RT）
- 错误率
- 并发数

#### 业务指标
- 订单量
- 注册用户数
- 活跃用户数

### 6.2 告警策略

#### 告警级别
- **紧急**：服务不可用
- **严重**：性能严重下降
- **警告**：性能下降
- **信息**：需要关注

#### 告警规则示例

| 指标 | 阈值 | 级别 | 通知方式 |
|------|------|------|---------|
| CPU 使用率 | > 90% | 紧急 | 短信+电话 |
| 错误率 | > 5% | 严重 | 短信 |
| 响应时间 | > 3s | 警告 | 邮件+钉钉 |
| 磁盘使用率 | > 90% | 警告 | 邮件 |

### 6.3 故障自愈

#### 自动恢复策略
- 实例故障 → 自动替换
- 服务异常 → 自动重启
- 资源不足 → 自动扩容

**配置示例：**
```yaml
# Kubernetes 自动扩缩容
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: app-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: app
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
```

## 七、高可用架构设计实战

### 7.1 典型 Web 应用高可用架构

```
用户
  ↓
DNS（多节点）
  ↓
CDN（多节点）
  ↓
WAF（多节点）
  ↓
SLB（主备，跨可用区）
  ↓
┌─────────────┐
│     AZ1     │    │     AZ2     │
│             │    │             │
│   ECS集群    │ ←→ │   ECS集群    │
│  (Web/App)  │    │  (Web/App)  │
│             │    │             │
│   Redis     │ ←→ │   Redis     │
│   (主备)    │    │   (主备)    │
│             │    │             │
│   RDS       │ ←→ │   RDS       │
│   (主备)    │    │   (主备)    │
└─────────────┘    └─────────────┘
         ↓                ↓
      OSS（多副本）
```

**可用性：**
- SLB：99.95%
- ECS：99.9% × 集群
- RDS：99.95%
- Redis：99.9%
- **整体可用性：约 99.9%**

### 7.2 微服务高可用架构

```
                    用户
                     ↓

                     ↓
            ┌────────┴────────┐
            ↓                 ↓
          服务A              服务B
            ↓                 ↓
      ┌────┴────┐         ┌───┴───┐
```
