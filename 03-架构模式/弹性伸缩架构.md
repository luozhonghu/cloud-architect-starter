# 架构模式 - 弹性伸缩架构

## 一、弹性伸缩概述

### 1.1 什么是弹性伸缩？

弹性伸缩（Auto Scaling）是指根据业务负载自动调整计算资源的能力，包括：
- **自动扩容**：负载增加时自动增加资源
- **自动缩容**：负载降低时自动减少资源
- **按需付费**：只使用需要的资源，降低成本

### 1.2 弹性伸缩架构

```
正常状态：
┌──────────┐
│ 用户流量  │
└────┬─────┘
     ↓
┌──────────┐
│   SLB    │
└────┬─────┘
     ↓
┌──────────┐
│ 2台ECS   │
└──────────┘

流量增加：
┌──────────┐
│ 用户流量  │
└────┬─────┘
     ↓
┌──────────┐
│   SLB    │
└────┬─────┘
     ↓
┌──────────┐
│ 10台ECS  │ ← 自动扩容
└──────────┘

流量恢复：
┌──────────┐
│ 用户流量  │
└────┬─────┘
     ↓
┌──────────┐
│   SLB    │
└────┬─────┘
     ↓
┌──────────┐
│ 2台ECS   │ ← 自动缩容
└──────────┘
```

### 1.3 弹性伸缩优势

| 优势 | 说明 |
|------|------|
| **成本优化** | 按需使用，避免资源浪费 |
| **性能保障** | 自动应对流量高峰 |
| **自动化** | 无需人工干预 |
| **高可用** | 故障时自动替换实例 |
| **灵活部署** | 支持多种伸缩策略 |

## 二、弹性伸缩类型

### 2.1 定时伸缩

#### 规律性业务
```
场景：电商每天晚高峰（20:00-23:00）

配置：
- 00:00-19:00：2台实例
- 20:00-23:00：10台实例
- 23:00-24:00：2台实例
```

**阿里云配置：**
```bash
# 创建定时任务
aliyun ess CreateScheduledTask \
  --RegionId cn-hangzhou \
  --ScheduledTaskName peak-hours-scaling \
  --ScheduledAction ess-xxxxx \
  --RecurrenceType Daily \
  --RecurrenceValue "20:00" \
  --LaunchTime "2024-01-01T20:00Z" \
  --ScalingRuleId srule-xxxxx
```

### 2.2 动态伸缩

#### 基于指标
```
指标：
- CPU 使用率 > 70% → 扩容
- CPU 使用率 < 30% → 缩容
- 内存使用率 > 80% → 扩容
- QPS > 1000 → 扩容
```

**阿里云配置：**
```bash
# 创建伸缩规则
aliyun ess CreateScalingRule \
  --RegionId cn-hangzhou \
  --ScalingGroupId ess-xxxxx \
  --ScalingRuleName cpu-scale-up \
  --AdjustmentType QuantityChangeInCapacity \
  --AdjustmentValue 2 \
  --Cooldown 300

# 创建告警任务
aliyun ess CreateAlarmTask \
  --RegionId cn-hangzhou \
  --AlarmTaskName cpu-alarm \
  --ScalingRuleId srule-xxxxx \
  --MetricType system_cpu_utilization \
  --Threshold 70 \
  --ComparisonOperator GreaterThanOrEqualToThreshold
```

### 2.3 预测伸缩

#### 智能预测
```
基于历史流量预测：
- 学习历史流量模式
- 预测未来流量趋势
- 提前准备资源
```

**阿里云配置：**
```bash
# 开启预测任务
aliyun ess EnablePredictiveScalingRule \
  --RegionId cn-hangzhou \
  --PredictiveScalingRuleId pssr-xxxxx \
  --PredictiveScalingBehavior PredictAndScale \
  --PredictiveValueBehavior PredictiveValueUpper
```

### 2.4 手动伸缩

#### 临时调整
```
场景：突发活动，临时增加资源

操作：
- 手动调整最小实例数
- 立即执行伸缩规则
- 临时覆盖策略
```

## 三、弹性伸缩策略

### 3.1 扩容策略

#### 立即扩容
```json
{
  "strategy": "immediate",
  "trigger": {
    "cpu_usage": "> 80%",
    "duration": "> 5分钟"
  },
  "action": {
    "add_instances": 5,
    "cooldown": "10分钟"
  }
}
```

#### 渐进扩容
```json
{
  "strategy": "gradual",
  "steps": [
    {
      "add_instances": 2,
      "wait": "5分钟"
    },
    {
      "check": "cpu_usage > 75%",
      "add_instances": 3,
      "wait": "5分钟"
    }
  ]
}
```

### 3.2 缩容策略

#### 保守缩容
```json
{
  "strategy": "conservative",
  "trigger": {
    "cpu_usage": "< 30%",
    "duration": "> 30分钟"
  },
  "action": {
    "remove_instances": 1,
    "cooldown": "30分钟"
  }
}
```

#### 激进缩容
```json
{
  "strategy": "aggressive",
  "trigger": {
    "cpu_usage": "< 20%",
    "duration": "> 15分钟"
  },
  "action": {
    "remove_instances": 2,
    "cooldown": "10分钟"
  }
}
```

### 3.3 冷却时间

```
扩容冷却：10分钟
- 避免频繁扩容
- 给新实例预热时间

缩容冷却：30分钟
- 避免频繁缩容
- 确保负载稳定
```

### 4.4 Kubernetes HPA

#### 部署应用
```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: myapp
spec:
  replicas: 2
  selector:
    matchLabels:
      app: myapp
  template:
    metadata:
      labels:
        app: myapp
    spec:
      containers:
      - name: myapp
        image: myapp:1.0
        ports:
        - containerPort: 8080
        resources:
          requests:
            cpu: "500m"
            memory: "512Mi"
          limits:
            cpu: "1000m"
            memory: "1Gi"
```

#### 配置 HPA
```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: myapp-hpa
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: myapp
  minReplicas: 2
  maxReplicas: 10
  metrics:
  - type: Resource
    resource:
      name: cpu
      target:
        type: Utilization
        averageUtilization: 70
  - type: Resource
    resource:
      name: memory
      target:
        type: Utilization
        averageUtilization: 80
  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
      - type: Percent
        value: 100
        periodSeconds: 60
      - type: Pods
        value: 4
        periodSeconds: 60
      selectPolicy: Max
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
      - type: Percent
        value: 50
        periodSeconds: 120
```

#### 创建 HPA
```bash
kubectl apply -f hpa.yaml

# 查看 HPA
```

## 七、适用场景

### ✅ 适合弹性伸缩的场景

| 场景 | 说明 |

| 场景 | 说明 |

| **特殊硬件** | GPU等特殊资源 |

## 八、实战案例

### 案例：电商大促弹性伸缩

#### 需求
- 日常流量：1000 QPS，2 台实例
- 大促高峰：10000 QPS，需要自动扩容到 20 台
- 大促结束后自动缩容到 2 台

#### 配置方案
```json
{
  "scaling_group": {
    "name": "ecommerce-scaling",
    "min_size": 2,
    "max_size": 20,
    "default_capacity": 2
  },
  "scaling_rules": [
    {
      "name": "scale-up-quick",
      "condition": "cpu_usage > 80% AND duration > 2min",
      "action": {
        "add_instances": 5,
        "cooldown": 300
      }
    },
    {
      "name": "scale-down-slow",
      "condition": "cpu_usage < 20% AND duration > 30min",
      "action": {
        "remove_instances": 1,
        "cooldown": 1800
      }
    }
  ],
  "scheduled_tasks": [
    {
      "name": "promo-pre-scale",
      "time": "2024-01-11T00:00:00Z",
      "action": "scale_to_10"
    },
    {
      "name": "promo-end-scale",
      "time": "2024-01-12T00:00:00Z",
      "action": "scale_to_2"
    }
  ]
}
```

#### 成本对比
```
固定部署（20台）：
成本：20 × ¥200 × 30 = ¥120,000/月

弹性伸缩：
成本：(2 × ¥200 × 30) + (10 × ¥200 × 2) = ¥16,000/月

节省：¥104,000/月（86.7%）
```

## 九、总结

### 关键要点

1. **选择合适策略**：定时、动态、预测
2. **设置合理阈值**：避免频繁伸缩
3. **实例预热**：确保新实例可用
4. **优雅下线**：避免请求失败
5. **监控告警**：及时发现问题
6. **成本优化**：混合实例、合理缩容

### 弹性伸缩 ≠ 魔法

弹性伸缩能帮助应对流量波动，但需要：
- 合理配置
- 持续优化
- 监控告警
- 业务配合

**记住：弹性伸缩是成本和性能的平衡，找到合适的策略是关键！**
