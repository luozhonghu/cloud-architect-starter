# 核心服务 - 监控服务

## 一、监控服务概述

### 1.1 什么是云监控？

云监控（Cloud Monitor）是云厂商提供的全方位监控解决方案，包括：
- **基础监控**：CPU、内存、磁盘、网络
- **应用监控**：APM（应用性能监控）、链路追踪
- **业务监控**：订单量、注册数、活跃用户等业务指标
- **日志监控**：日志采集、分析、告警
- **安全监控**：安全事件、攻击检测

## 二、阿里云监控服务

### 2.1 云监控（Cloud Monitor）

#### 主机监控
```bash
# 安装监控插件
wget https://cms-agent-cn-hangzhou.oss-cn-hangzhou.aliyuncs.com/monitor-packages/cms-agent-1.3.0.tar.gz
tar -zxvf cms-agent-1.3.0.tar.gz
cd cms-agent-1.3.0
./install.sh

# 查看监控数据
aliyun cms DescribeMetricList \
  --Namespace acs_ecs_dashboard \
  --MetricName cpu_total \
  --Dimensions '{"instanceId":"i-xxxxx"}' \
  --StartTime 2024-01-01T00:00:00Z \
  --EndTime 2024-01-01T23:59:59Z
```

#### 监控指标
| 指标分类 | 指标名称 | 说明 | 阈值建议 |
|---------|---------|------|---------|
| **CPU** | cpu_total | 总 CPU 使用率 | > 80% |
| **内存** | memory_usedutilization | 内存使用率 | > 85% |
| **磁盘** | diskusage_utilization | 磁盘使用率 | > 90% |
| **网络** | intranet_in_rate_bytes | 内网入流量 | > 预期 |
| **网络** | intranet_out_rate_bytes | 内网出流量 | > 预期 |
| **负载** | load_1m | 1分钟平均负载 | > CPU 核心数 |

#### 告警规则
```bash
# 创建告警规则
aliyun cms CreateMetricRuleTemplate \
  --Name cpu-alert \
  --Namespace acs_ecs_dashboard \
  --MetricName cpu_total \
  --Threshold 80 \
  --ComparisonOperator GreaterThanOrEqualToThreshold \
  --EvaluationCount 3 \
  --Statistics Average \
  --Period 60

# 绑定告警规则到实例
aliyun cms ApplyMetricRuleTemplate \
  --TemplateId template-xxxxx \
  --Targets '[{"instanceId":"i-xxxxx"}]'
```

#### 应用接入


```yaml
# application.yml
spring:
  application:
    name: myapp

# ARMS 配置
arms:
  licenseKey: your-license-key
  appName: ${spring.application.name}
  # 采样率（100% 采样会消耗更多性能）
  sampling: 10
  # 是否启用监控
  enabled: true
```

#### 应用性能指标
```java
// 自定义指标埋点
@RestController
public class UserController {
    
    @Autowired
    private ArmsService armsService;
    
    @GetMapping("/api/user/{id}")
    public User getUser(@PathVariable Long id) {
        // 记录方法调用
        armsService.recordApiMethod("getUser");
        
        // 记录业务指标
        armsService.recordBusinessMetric("user_query", 1);
        
        // 记录响应时间
        long start = System.currentTimeMillis();
        try {
            User user = userService.findById(id);
            armsService.recordResponseTime("getUser", System.currentTimeMillis() - start);
            return user;
        } catch (Exception e) {
            armsService.recordError("getUser", e);
            throw e;
        }
    }
}
```

### 2.3 日志服务（SLS）

#### 日志采集
```yaml
# Logtail 配置
inputs:
  - type: file_log
    log_path: /opt/app/logs/
    file_pattern: "*.log"
    topic_format: default
    filter_regex:
      - ERROR
      - WARN

processors:
  - type: processor_regex
    source_key: content
    regex: '(\d{4}-\d{2}-\d{2} \d{2}:\d{2}:\d{2}) \[(\w+)\] (.*)'
    keys:
      - timestamp
      - level
      - message
```

#### 日志查询
```bash
# 查询日志
aliyun logs GetLogs \
  --ProjectName myapp-logs \
```

#### 告警规则
```yaml
# alerting.yml
groups:
  - name: application
    interval: 30s
    rules:
      # CPU 告警
      - alert: HighCPUUsage
        expr: rate(process_cpu_seconds_total[5m]) > 0.8
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High CPU usage on {{ $labels.instance }}"
          description: "CPU usage is above 80% for 5 minutes"
      
      # 内存告警
      - alert: HighMemoryUsage
        expr: jvm_memory_used_bytes{area="heap"} / jvm_memory_max_bytes{area="heap"} > 0.85
        for: 5m
        labels:
          severity: warning
        annotations:
          summary: "High memory usage on {{ $labels.instance }}"
          description: "Heap memory usage is above 85%"
      
      # 错误率告警
      - alert: HighErrorRate
        expr: rate(http_server_requests_seconds_count{status=~"5.."}[5m]) / rate(http_server_requests_seconds_count[5m]) > 0.05
        for: 5m
        labels:
          severity: critical
        annotations:
          summary: "High error rate on {{ $labels.instance }}"
          description: "Error rate is above 5%"
```

## 五、监控最佳实践

### 5.1 监控指标设计

#### 黄金指标
| 指标类型 | 说明 | 示例 |
|---------|------|------|
| **延迟** | 请求响应时间 | P50、P95、P99 |
| **流量** | 请求数量 | QPS、TPS |
| **错误** | 错误率 | HTTP 5xx、业务错误 |
| **饱和度** | 资源使用率 | CPU、内存、磁盘 |

#### 监控分层
```
基础设施层
  ↓ CPU、内存、磁盘、网络
容器层
  ↓ Pod、Container
应用层
  ↓ QPS、RT、错误率
业务层
  ↓ 订单量、注册数、GMV
```

### 5.2 告警策略

#### 告警级别
| 级别 | 阈值 | 通知方式 | 响应时间 |
|------|------|---------|---------|
| **紧急** | 服务不可用 | 电话 + 短信 | 5分钟 |
| **严重** | 性能严重下降 | 短信 + 钉钉 | 15分钟 |
| **警告** | 性能下降 | 邮件 + 钉钉 | 30分钟 |
| **信息** | 需要关注 | 邮件 | 1小时 |

#### 告警降噪
```yaml
# 告警分组
group_by: [alertname, cluster]

# 告警抑制
inhibit_rules:
  - source_match:
      severity: 'critical'
    target_match:
      severity: 'warning'
    equal: ['alertname', 'cluster']

# 告警静默
silences:
  - id: maintenance
    matchers:
      - name: alertname
        value: HighCPUUsage
    startsAt: 2024-01-01T00:00:00Z
    endsAt: 2024-01-01T06:00:00Z
```

## 七、总结

### 关键要点

1. **分层监控**：基础设施 → 应用 → 业务
2. **黄金指标**：延迟、流量、错误、饱和度
3. **告警分级**：紧急、严重、警告、信息
4. **可视化**：监控大盘、实时告警
5. **闭环处理**：告警 → 定位 → 修复 → 复盘

### 监控 ≠ 告警

监控的目的是：
- 及时发现问题
- 快速定位问题
- 持续优化系统

**记住：好的监控体系能让你在问题发生前就发现它！**
