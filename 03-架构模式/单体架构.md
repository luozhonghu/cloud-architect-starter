# 架构模式 - 单体架构

## 一、单体架构概述

### 1.1 什么是单体架构？

单体架构（Monolithic Architecture）是指将整个应用程序作为一个单一的单元进行开发、部署和运行的架构模式。所有功能模块（用户管理、订单管理、支付管理等）都打包在一个应用程序中，共享同一个数据库。

### 1.2 架构特点

```
┌─────────────────────────────────────┐
│         Web 应用程序（单体）          │
│                                     │
│  ┌─────────┬─────────┬─────────┐   │
│  │ 用户模块 │ 订单模块 │ 支付模块 │   │
│  └─────────┴─────────┴─────────┘   │
│         ↓         ↓         ↓     │
│  ┌─────────────────────────────┐   │
│  │      共享业务逻辑层         │   │
│  └─────────────────────────────┘   │
│         ↓                          │
│  ┌─────────────────────────────┐   │
│  │       数据访问层（DAO）     │   │
│  └─────────────────────────────┘   │
│         ↓                          │
│  ┌─────────────────────────────┐   │
│  │      数据库（单个DB）        │   │
│  └─────────────────────────────┘   │
└─────────────────────────────────────┘
```

### 1.3 技术栈示例

| 层级 | 技术选择 |
|------|---------|
| 前端 | Vue.js / React / Angular |
| 后端 | Java Spring Boot / Python Django / Node.js Express |
| 数据库 | MySQL / PostgreSQL |
| 缓存 | Redis |
| 部署 | Tomcat / Nginx / Docker |

### 1.4 单体架构优缺点

#### ✅ 优点

| 优点 | 说明 |
|------|------|
| **简单易用** | 架构简单，易于理解和开发 |
| **开发快速** | 无需考虑分布式问题，开发效率高 |
| **测试容易** | 端到端测试简单 |
| **部署方便** | 只需部署一个应用包 |
| **调试简单** | 所有代码在一个项目中，调试方便 |
| **事务一致** | 本地事务保证数据一致性 |

#### ❌ 缺点

| 缺点 | 说明 |
|------|------|
| **扩展困难** | 只能整体扩展，无法针对特定模块扩展 |
| **耦合度高** | 模块间耦合严重，修改影响范围大 |
| **技术栈受限** | 所有模块使用相同技术栈 |
| **可靠性差** | 一个模块的问题可能导致整个应用崩溃 |
| **部署风险高** | 部署需要停机或灰度，风险较大 |
| **团队协作难** | 大团队协作困难，代码冲突多 |

## 二、单体架构在云上的实现

### 2.1 基础架构

#### 单实例部署
```
用户 → CDN → SLB → 单个 ECS（Web 应用） → RDS
```

**特点：**
- 最简单的部署方式
- 成本最低
- 可用性较差（单点故障）

**适用场景：**
- 小型应用
- 内部系统
- 开发测试环境

```
用户 → CDN → SLB → ECS-1 → RDS
```

**特点：**
- 多实例部署
- 高可用
- 可水平扩展

**适用场景：**

### 2.2 部署方式

#### 传统部署
```bash

# 2. 上传到服务器

kill -9 $(ps aux | grep app.jar | grep -v grep | awk '{print $2}')
```

## 三、单体架构的设计最佳实践

### 3.1 分层架构

虽然应用是单体的，但仍然应该采用分层架构，提高代码的可维护性。

```
┌─────────────────────────────────┐
│      表示层（Controller）        │  ← 处理 HTTP 请求
├─────────────────────────────────┤
│      业务逻辑层（Service）       │  ← 业务逻辑处理
├─────────────────────────────────┤
│      数据访问层（DAO/Repository）│ ← 数据库操作
├─────────────────────────────────┤
│      数据库层（Database）         │  ← 数据持久化
└─────────────────────────────────┘
```

#### Spring Boot 分层示例

```java
// Controller 层
@RestController
@RequestMapping("/api/users")
public class UserController {
    
    @Autowired
    private UserService userService;
    
    @GetMapping("/{id}")
    public UserDTO getUser(@PathVariable Long id) {
        return userService.getUserById(id);
    }
}

// Service 层
@Service
public class UserService {
    
    @Autowired
    private UserRepository userRepository;
    
    public UserDTO getUserById(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        return convertToDTO(user);
    }
    
    private UserDTO convertToDTO(User user) {
        // 转换逻辑
    }
}

// Repository 层
@Repository
public interface UserRepository extends JpaRepository<User, Long> {
}
```

### 3.2 模块化设计

虽然代码在一个项目中，但应该按模块组织，降低耦合度。

```
src/main/java/com/company/
├── common/              # 公共模块
│   ├── exception/
│   ├── util/
│   └── constant/
├── user/               # 用户模块
│   ├── controller/
│   ├── service/
│   ├── repository/
│   └── model/
├── order/              # 订单模块
│   ├── controller/
│   ├── service/
│   ├── repository/
│   └── model/
└── payment/            # 支付模块
    ├── controller/
    ├── service/
    ├── repository/
    └── model/
```

### 3.3 依赖注入

使用依赖注入降低模块间耦合。

```java
// 接口定义
public interface PaymentService {
    PaymentResult pay(PaymentRequest request);
}

// 实现
@Service("alipayService")
public class AlipayService implements PaymentService {
    public PaymentResult pay(PaymentRequest request) {
        // 支付宝支付实现
    }
}

@Service("wechatpayService")
public class WechatpayService implements PaymentService {
    public PaymentResult pay(PaymentRequest request) {
        // 微信支付实现
    }
}

// 使用
@RestController
public class PaymentController {
    
    @Autowired
    private Map<String, PaymentService> paymentServices;
    
    public PaymentResult pay(String type, PaymentRequest request) {
        PaymentService service = paymentServices.get(type + "Service");
        return service.pay(request);
    }
}
```

## 四、单体架构的扩展策略

### 4.1 垂直扩展（Scale Up）

**方案：**
```
升级硬件配置
  ↓
增加 CPU、内存、磁盘
  ↓
提升单机性能
```

**操作步骤：**
```bash
# 阿里云 ECS 升级配置
aliyun ecs ModifyInstanceSpec \
  --InstanceId i-xxxxx \
  --InstanceType ecs.g6.xlarge \
  --AutoPay true
```

**优缺点：**
- ✅ 简单直接
- ✅ 无需修改代码
- ❌ 有上限
- ❌ 成本高

### 4.2 水平扩展（Scale Out）

**方案：**
```
增加实例数量
  ↓
部署多个相同实例
  ↓
使用负载均衡分发流量
```

**操作步骤：**
```bash
# 1. 创建自定义镜像
aliyun ecs CreateImage \
  --InstanceId i-xxxxx \
  --ImageName myapp-image

# 2. 使用镜像创建多个实例
aliyun ecs CreateInstance \
  --ImageId m-xxxxx \
  --InstanceType ecs.t6-c1m2.small \
  --Amount 3

# 3. 配置负载均衡
aliyun slb AddBackendServers \
  --LoadBalancerId lb-xxxxx \
  --BackendServers "[{\"ServerId\":\"i-1\"},{\"ServerId\":\"i-2\"},{\"ServerId\":\"i-3\"}]"
```

**优缺点：**
- ✅ 理论无上限
- ✅ 成本可控
- ❌ 需要无状态设计
- ❌ 需要负载均衡

### 4.3 数据库扩展

#### 读写分离
```
应用 → 读写分离代理 → 主库（写）
                    → 从库1（读）
                    → 从库2（读）
```

#### 分库分表
```
用户表 → user_0, user_1, user_2, ...
订单表 → order_0, order_1, order_2, ...
```

## 六、单体架构向微服务演进

### 6.1 演进时机

#### 何时考虑拆分？
| 指标 | 阈值 |
|------|------|
| 代码行数 | > 50 万行 |
| 团队人数 | > 20 人 |
| 部署时间 | > 30 分钟 |
| 构建时间 | > 10 分钟 |
| 单个模块变更频率 | 高 |

### 6.2 演进路径

```
单体架构
   ↓
模块化单体
   ↓
微服务架构
```

#### 第一阶段：模块化单体
- 按业务模块划分代码
- 使用接口隔离模块
- 保持单体部署

#### 第二阶段：服务拆分
- 识别业务边界
- 按边界拆分服务
- 共享数据库

#### 第三阶段：完全微服务
- 每个服务独立数据库
- 服务间通过 API 通信
- 独立部署和扩展

## 八、适用场景

### ✅ 适合单体架构的场景

1. **初创项目**：快速开发和迭代
2. **小型应用**：用户量 < 10 万
3. **内部系统**：无需高并发
4. **MVP 验证**：快速验证想法
5. **小团队**：团队人数 < 10 人

### ❌ 不适合单体架构的场景

1. **大型应用**：用户量 > 100 万
2. **高并发场景**：QPS > 1 万
3. **复杂业务**：多个业务线独立发展
4. **大团队**：团队人数 > 20 人
5. **需要独立部署**：不同模块独立上线

## 十、总结

单体架构虽然简单，但并不是"过时"的架构。对于大多数项目，单体架构仍然是最合适的选择。

### 关键要点

1. **从简单开始**：不要一开始就上微服务
2. **模块化设计**：即使单体也要模块化
3. **关注可扩展性**：为未来演进做好准备
4. **完善监控**：做好监控和日志
5. **按需演进**：根据业务需求决定是否拆分

### 何时拆分？

**拆分单体架构的时机：**
- 代码太大，难以维护
- 部署太慢，影响发布
- 团队太大，协作困难
- 不同模块需要独立扩展
- 不同模块需要不同技术栈

### 保留单体的理由

**保留单体架构的理由：**
- 团队规模小
- 业务复杂度低
- 快速迭代需求
- 成本控制需求

**单体架构 + 良好的模块化设计 = 最佳起步方案！**
