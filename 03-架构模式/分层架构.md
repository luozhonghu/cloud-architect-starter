# 架构模式 - 分层架构

## 一、分层架构概述

### 1.1 什么是分层架构？

分层架构（Layered Architecture）是最常见、最经典的软件架构模式。它将应用程序划分为多个层次，每一层只负责特定的功能，上层通过接口调用下层的服务。

### 1.2 经典分层

```
┌─────────────────────────────────────┐
│      表示层（Presentation Layer）    │  ← 处理用户交互
├─────────────────────────────────────┤
│      业务逻辑层（Business Layer）    │  ← 处理业务逻辑
├─────────────────────────────────────┤
│      数据访问层（Data Access Layer） │  ← 访问数据库
├─────────────────────────────────────┤
│      数据存储层（Database Layer）    │  ← 数据持久化
└─────────────────────────────────────┘
```

### 1.3 分层原则

| 原则 | 说明 |
|------|------|
| **单向依赖** | 上层依赖下层，下层不依赖上层 |
| **层间隔离** | 每层只关注自己的职责 |
| **接口稳定** | 层间通过接口通信 |
| **松耦合** | 层与层之间低耦合 |

```
                  ↓

                  ↓
```


#### Controller 层
```java
@RestController
@RequestMapping("/api/users")
public class UserController {
    
    @Autowired
    private UserService userService;
    
    @GetMapping("/{id}")
    public Result<UserDTO> getUser(@PathVariable Long id) {
        UserDTO user = userService.getUserById(id);
        return Result.success(user);
    }
    
    @PostMapping
    public Result<UserDTO> createUser(@RequestBody @Valid UserCreateDTO dto) {
        UserDTO user = userService.createUser(dto);
        return Result.success(user);
    }
    
    @PutMapping("/{id}")
    public Result<UserDTO> updateUser(@PathVariable Long id, 
                                      @RequestBody @Valid UserUpdateDTO dto) {
        UserDTO user = userService.updateUser(id, dto);
        return Result.success(user);
    }
    
    @DeleteMapping("/{id}")
    public Result<Void> deleteUser(@PathVariable Long id) {
        userService.deleteUser(id);
        return Result.success();
    }
}
```

#### Service 层
```java
@Service
@Transactional
public class UserService {
    
    @Autowired
    private UserRepository userRepository;
    
    @Autowired
    private UserMapper userMapper;
    
    public UserDTO getUserById(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        return userMapper.toDTO(user);
    }
    
    public UserDTO createUser(UserCreateDTO dto) {
        // 1. 验证数据
        validateUserCreateDTO(dto);
        
        // 2. 检查用户是否已存在
        if (userRepository.existsByUsername(dto.getUsername())) {
            throw new UsernameAlreadyExistsException(dto.getUsername());
        }
        
        // 3. 创建用户
        User user = userMapper.toEntity(dto);
        user.setPassword(encodePassword(dto.getPassword()));
        user = userRepository.save(user);
        
        // 4. 返回结果
        return userMapper.toDTO(user);
    }
    
    public UserDTO updateUser(Long id, UserUpdateDTO dto) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        
        // 更新字段
        if (dto.getEmail() != null) {
            user.setEmail(dto.getEmail());
        }
        if (dto.getPhone() != null) {
            user.setPhone(dto.getPhone());
        }
        
        user = userRepository.save(user);
        return userMapper.toDTO(user);
    }
    
    public void deleteUser(Long id) {
        User user = userRepository.findById(id)
            .orElseThrow(() -> new UserNotFoundException(id));
        userRepository.delete(user);
    }
    
    private void validateUserCreateDTO(UserCreateDTO dto) {
        if (StringUtils.isEmpty(dto.getUsername())) {
            throw new InvalidParameterException("username不能为空");
        }
        if (StringUtils.isEmpty(dto.getPassword())) {
            throw new InvalidParameterException("password不能为空");
        }
    }
    
    private String encodePassword(String rawPassword) {
        // 密码加密逻辑
        return BCrypt.hashpw(rawPassword, BCrypt.gensalt());
    }
}
```

#### Mapper 转换
```java
@Component
public class UserMapper {
    
    public UserDTO toDTO(User user) {
        if (user == null) {
            return null;
        }
        UserDTO dto = new UserDTO();
        dto.setId(user.getId());
        dto.setUsername(user.getUsername());
        dto.setEmail(user.getEmail());
        dto.setPhone(user.getPhone());
        dto.setCreateTime(user.getCreateTime());
        dto.setUpdateTime(user.getUpdateTime());
        return dto;
    }
    
    public User toEntity(UserCreateDTO dto) {
        if (dto == null) {
            return null;
        }
        User user = new User();
        user.setUsername(dto.getUsername());
        user.setPassword(dto.getPassword());
        user.setEmail(dto.getEmail());
        user.setPhone(dto.getPhone());
        return user;
    }
}
```

## 四、分层架构的优势

### 4.1 关注点分离

| 层级 | 关注点 | 示例 |
|------|--------|------|
| **表示层** | 用户交互 | 处理 HTTP 请求/响应 |
| **业务层** | 业务逻辑 | 订单创建、用户验证 |
| **数据层** | 数据访问 | 数据库 CRUD 操作 |

### 4.2 易于维护

```
修改 UI → 只改表示层
修改业务 → 只改业务层
修改数据 → 只改数据层
```

### 4.3 易于测试

```java
// 单元测试：Service 层
@SpringBootTest
public class UserServiceTest {
    
    @Autowired
    private UserService userService;
    
    @MockBean
    private UserRepository userRepository;
    
    @Test
    public void testGetUserById() {
        // Given
        User user = new User();
        user.setId(1L);
        user.setUsername("test");
        when(userRepository.findById(1L)).thenReturn(Optional.of(user));
        
        // When
        UserDTO result = userService.getUserById(1L);
        
        // Then
        assertNotNull(result);
        assertEquals("test", result.getUsername());
    }
}
```

### 4.4 易于扩展

```
添加新的前端 → 只需修改表示层
添加新的数据源 → 只需修改数据层
```

## 五、分层架构的最佳实践

### 5.1 层间依赖规则

```
✅ 正确：
Controller → Service → Repository → Database

❌ 错误：
Controller → Repository (跳过Service)
Service → Controller (反向依赖)
```

### 5.2 异常处理

#### 全局异常处理
```java
@RestControllerAdvice
public class GlobalExceptionHandler {
    
    @ExceptionHandler(UserNotFoundException.class)
    public ResponseEntity<Result<Void>> handleUserNotFound(UserNotFoundException e) {
        return ResponseEntity.status(HttpStatus.NOT_FOUND)
            .body(Result.error(e.getMessage()));
    }
    
    @ExceptionHandler(MethodArgumentNotValidException.class)
    public ResponseEntity<Result<Void>> handleValidationException(
            MethodArgumentNotValidException e) {
        String message = e.getBindingResult()
            .getFieldErrors()
            .stream()
            .map(FieldError::getDefaultMessage)
            .collect(Collectors.joining(", "));
        return ResponseEntity.badRequest()
            .body(Result.error(message));
    }
    
    @ExceptionHandler(Exception.class)
    public ResponseEntity<Result<Void>> handleException(Exception e) {
        log.error("系统异常", e);
        return ResponseEntity.status(HttpStatus.INTERNAL_SERVER_ERROR)
            .body(Result.error("系统异常，请稍后重试"));
    }
}
```

## 七、分层架构 vs 其他架构

### 7.1 与单体架构对比

| 维度 | 分层架构 | 单体架构（不分层） |
|------|---------|------------------|
| **代码组织** | 按层次组织 | 按功能组织 |
| **可维护性** | 高 | 低 |
| **可测试性** | 高 | 低 |
| **学习成本** | 中 | 低 |
| **开发速度** | 中 | 快 |

### 7.2 与微服务架构对比

| 维度 | 分层架构 | 微服务架构 |
|------|---------|-----------|
| **部署** | 整体部署 | 独立部署 |
| **复杂度** | 低 | 高 |
| **扩展性** | 垂直扩展 | 水平扩展 |
| **团队协作** | 大团队 | 小团队 |
| **成本** | 低 | 高 |

## 八、适用场景

### ✅ 适合分层架构的场景

| 场景 | 说明 |
|------|------|
| **中小型应用** | 代码 < 50 万行 |
| **传统企业应用** | 管理系统、ERP等 |
| **前后端分离** | Web + API |
| **团队规模适中** | 10-20 人 |
| **需要快速开发** | MVP、原型 |

### ❌ 不适合分层架构的场景

| 场景 | 说明 |
|------|------|
| **超大型应用** | 代码 > 100 万行 |
| **高并发场景** | QPS > 10 万 |
| **多业务线** | 需要独立部署 |
| **微服务架构** | 已拆分为微服务 |

## 十、总结

### 关键要点

1. **分层是基础**：所有架构都应该分层
2. **单向依赖**：上层依赖下层，不要反向依赖
3. **职责清晰**：每层只做自己的事
4. **接口稳定**：层间通过接口通信
5. **易于测试**：每层都可以独立测试
6. **适度分层**：不要过度分层

### 演进路径

```
简单分层
   ↓
完整分层
   ↓
DDD分层
   ↓
微服务（按域拆分）
```

### 最佳实践

**记住：分层架构是软件架构的基础，无论单体还是微服务，都应该遵循分层原则！**
